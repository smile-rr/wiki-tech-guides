memory_compact_harness

Short-term Memory 的生成与更新
每条消息落库后（t1 同步）
user/assistant 消息写入 messages 表同时，同步写入 Redis（RPUSH）。这是 working memory 的实时更新，每轮对话都发生，延迟 <1ms，不需要额外 LLM call。
工具调用完成后（t3/t5 同步）
tool_use + tool_result 写入 messages 表的同时写入 Redis。agent 下一轮能"记住"工具执行结果，靠的就是这两条 Redis 记录。
emit_state 工具调用后（每轮 LLM call 结束）
从 response 的 tool_use block 里提取 emit_state 的 input，写入 Redis agent_state:{session_id}。这是状态层的实时更新，下次 call 的 system[1] 从这里读。是应用代码提取，不是额外 LLM call。
Session 超阈值后（t9 异步，含 LLM call）
token_count 超过阈值（如 6000/8K window 的 75%）时触发 Compact，用一次额外 LLM call 生成 working memory 的压缩版，存入 session_context 表和 Redis。下文详述。
Long-term Memory 的生成时机（三个触发点）
触发点 1：Assistant 回复包含用户信息后（异步，含 LLM call）
用一个极轻量的 LLM call（判断是否包含值得记忆的内容）：
— 用户表达明确偏好：「我不喜欢经停」→ 触发
— 用户说明个人信息：「我的会员号是...」→ 触发
— 助手确认了重要决策：「好的，以后都优先国航」→ 触发
— 普通问答、闲聊 → 不触发，跳过
触发后再用一次 LLM call 提取结构化记忆，写入 user_memories + pgvector。
触发点 2：Session 结束时（异步，含 LLM call）
用户关闭对话或超时后，对整个 session 做一次"记忆蒸馏"，提取本次 session 产生的新偏好/事实/决策，增量写入 long-term memory。这是最稳的触发点，不影响在线延迟。
触发点 3：用户显式操作（同步，含 LLM call）
用户点击「记住这个」按钮，或说「以后都记住XXX」。立即触发记忆写入。这类记忆 importance 分值设最高（1.0），RAG 检索时优先返回。
Long-term memory 生成是昂贵操作，不能每轮都触发。 实践中用一个极小的分类模型（FastText/规则）先判断"这条对话是否含有值得记忆的内容"，命中率约 10-20%，只有命中才进入 LLM 提取流程。每次 LLM call 都生成 long-term memory 是常见的过度设计错误。
Compact 是什么：一次专用系统指令的额外 LLM Call
直接回答：是的，compact 是一次额外的 LLM call，有独立的 system prompt，目的是把当前 working memory（messages 历史）压缩成摘要，替换掉原始历史，让后续 call 的 token 消耗保持可控。

Compact 专用 system prompt（与业务 prompt 完全不同）
// compact_system_prompt.txt（专用，不含业务逻辑） COMPACT_SYSTEM = """ 你的任务是将以下对话历史压缩成结构化摘要。 输出必须包含以下字段（JSON格式）： - task: 用户的原始任务目标 - context: 关键背景信息（用户偏好、约束条件） - progress: 已完成的步骤列表 - tried: 已尝试过的方案及结果 - current_state: 当前所处阶段 - pending: 还未完成的子任务 - key_facts: 对话中确认的重要事实 要求： 1. 保留所有工具调用结果中的具体数据（航班号、价格、时间等） 2. 保留用户明确表达的偏好和拒绝 3. 删除思考过程、礼貌用语、重复内容 4. 输出必须让接手的 Agent 能无缝继续任务 """ // compact call 的 request 结构 compact_request = { "model": "qwen2.5-7b-instruct", // 可以用更小更快的模型 "max_tokens": 1024, "messages": [ { "role": "system", "content": COMPACT_SYSTEM }, { "role": "user", "content": "请压缩以下对话历史：\n\n" + format_messages(full_history) } ] // 注意：不带 tools，这是纯摘要任务 }
Compact 输出样例 → 成为下次 call 的 system[1]

Compact vs AutoCompact 的区别
Manual Compact（主动压缩）
应用代码在合适时机主动触发，时机可控，压缩前 context 还宽裕，摘要质量高。

推荐在 60-70% 阈值时触发，模型视野还清晰，能做高质量摘要。

可以带"压缩重点"参数：compress(focus="机票部分")
AutoCompact（被动压缩）
推理框架（vLLM 等）或应用在 context 接近上限时自动触发，通常 80-90% 阈值。

此时 context 已拥挤，模型处于"lost in middle"状态，摘要质量低于 Manual Compact。

是 safety net，不是首选策略。
Compact 有信息损失。 摘要是有损压缩，工具调用的原始 JSON、用户的措辞细节会丢失。实践中：保留工具结果里的具体数字（航班号、价格），丢弃思考过程和礼貌用语。关键决策数据宁可冗余也不能丢。
三个时代：关注点的根本性转移
1
Prompt Engineering
2022–2023 · 单次 call 时代
核心问题
怎么写 prompt 让模型给出想要的答案
工作内容
调 system prompt 措辞、few-shot 例子、CoT 指令、温度参数。一个问题发一次 call，收到回答就完事。
心智模型
模型是函数：输入 → 输出。工程师负责精心构造输入。
主要挑战
prompt 敏感性（换个词结果完全不同）、格式不稳定、幻觉
few-shot CoT 温度调参 角色扮演 格式约束
2
Context Management
2023–2024 · RAG + 多轮时代
核心问题
context window 是稀缺资源，怎么管理放什么进去
工作内容
RAG 向量检索、文档 chunking 策略、sliding window、prompt caching、多轮历史截断、知识库构建。开始有"哪些信息值得放进 context"的优先级思考。
心智模型
context window 是有限的工作台：精心挑选放上去的东西比精心措辞更重要。
主要挑战
lost-in-the-middle、检索召回率、token 成本、历史截断丢失关键信息
RAG 向量数据库 Chunking Reranker KV Cache 滑动窗口
3
Harness Engineering
2024–至今 · Agentic 时代
核心问题
LLM 是推理引擎，需要整套执行环境才能可靠运行
工作内容
设计整个"harness"（驾驭层）：
· 状态机：emit_state 追踪任务进度，不靠 messages 隐式传递
· 记忆架构：long/short memory 分层，各有生成时机和存储
· Compact 策略：主动压缩，而不是等 context 溢出
· 工具编排：决策流、降级方案、tool search、defer_loading
· 多模型路由：意图分类用 FastText，摘要用小模型，推理用大模型
· 可观测性：Langfuse traces，每次 LLM call 有 span，成本可审计
· 可靠性：工具失败重试、graceful degradation、人工干预点
心智模型
LLM 是不可靠的推理引擎，Harness 是让它可靠运行的基础设施。工程师的工作从"写 prompt"变成"设计执行环境"。
主要挑战
状态一致性、多步任务失败恢复、并发 session 隔离、成本与质量平衡
状态机 emit_state Compact Memory 分层 多模型路由 可观测性 降级策略 人工干预点
三个时代的"问题粒度"对比
// Prompt Engineering 时代的问题： "怎么让模型输出 JSON 格式而不是自然语言？" "加一句 'think step by step' 效果更好吗？" "few-shot 要放几个例子？" // Context Management 时代的问题： "检索出来的 chunk 质量太差，要改 chunk 大小还是改 embedding 模型？" "prompt cache 命中率只有 60%，是因为 system prompt 每次都变了？" "长对话后模型开始忘记前面的内容，滑动窗口要设多大？" // Harness Engineering 时代的问题： "查不到直飞时，agent 自动切中转是对的，但如何在中转也失败时优雅降级？" "compact 后模型丢失了用户 3 轮前确认的预算上限，摘要格式要改什么？" "emit_state 工具被模型遗漏调用了，需要在 app 层加强制检查逻辑？" "session 并发高时 Redis 的 agent_state key 有没有竞态问题？" "long-term memory 每次都触发导致成本翻 3 倍，分类器阈值要怎么调？"
Harness Engineering 的本质转变： 从"让模型表现更好"转向"让模型在不可靠的情况下仍然能可预期地运行"。工程师的职责从提示词设计师变成了 AI 系统的基础设施工程师——需要考虑状态管理、失败恢复、成本控制、可观测性，这些在 prompt engineering 时代根本不是问题。
