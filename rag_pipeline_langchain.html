2. 各阶段 LangChain Class 详解

2.1 Stage ① — Document Loaders（文档加载器）

将原始文件或数据源转换为 LangChain Document 对象（包含 page_content 正文 + metadata 元数据）。

Class 名称功能说明你项目里对应核心参数 / 重要区别
PyMuPDFLoaderPDF 加载，保留布局和表格结构技术文档、合规文件比 PyPDFLoader 快 5~10 倍，保留更多格式信息；推荐首选
PyPDFLoader基础 PDF 文字提取，每页独立 Document普通 PDF无布局信息，处理速度慢；简单场景可用
UnstructuredPDFLoaderPDF + OCR 支持扫描件扫描版文档依赖 unstructured 库；处理复杂 PDF 最强但最慢
CSVLoader每行转为一个 Document结构化数据、配置表source_column 参数指定 metadata 来源列；适合 DORA metrics 数据
UnstructuredExcelLoaderExcel 多 sheet 加载报表、数据字典自动处理合并单元格；需要 unstructured 库
ConfluenceLoader直接拉取 Confluence 页面（你项目核心）Confluence Q&A 项目space_key / page_ids 过滤；metadata 包含 page_title、space_key、url
DirectoryLoader批量加载目录下所有文件本地知识库批量导入glob 参数过滤文件类型，recursive=True 支持递归
WebBaseLoader抓取网页内容实时网页知识库基于 BeautifulSoup；JS 渲染页面需要 Playwright
NotionDBLoader加载 Notion 数据库页面Notion 知识库需要 Notion API token；支持 filter 参数
Confluence 项目实际用法：
ConfluenceLoader(url=CONF_URL, username=USER, api_key=KEY, space_key="TEAM")
离线索引架构中 Loader 只在批量 ingestion 时调用，不在用户查询路径——这正是你简历里"eliminating API rate limit bottlenecks"的核心逻辑：把拉取动作从"每次查询实时调"变为"定时批量拉"。

2.2 Stage ② — Text Splitters（文本分割器）

将 Document 切分为适合 embedding 的 chunk。chunk_size 控制每块大小，chunk_overlap 控制重叠保留上下文。

Class 名称功能说明你项目里对应核心参数 / 重要区别
RecursiveCharacterTextSplitter递归按分隔符分割（首选推荐）通用文本，Confluence 页面separators=["\n\n","\n"," "]，语义保留最好；大多数场景用这个
CharacterTextSplitter按单一字符分割简单文本只按一个 separator，chunk 边界可能切在句子中间
MarkdownTextSplitter按 Markdown 标题层级分割MD 格式文档保留标题结构，适合 Confluence Markdown 导出
TokenTextSplitter按 token 数精确控制精确控制 context 长度依赖 tiktoken；用于需要精确 token 计数的场景
HTMLHeaderTextSplitter按 HTML 标题层级分割Confluence HTML 导出metadata 保留标题层级（h1/h2/h3），检索结果更可解释；Confluence 场景最佳
SemanticChunker基于 embedding 相似度分割高语义一致性要求相邻句子 embedding 相似度低时切割；质量最高但速度慢
Confluence 最佳实践（两段式分割）：
第一步 HTMLHeaderTextSplitter（按 h1/h2 切大块，保留标题 metadata）→ 第二步 RecursiveCharacterTextSplitter（chunk_size=512, overlap=50，把大块进一步切小）
metadata 保留 page_title + space_key + heading，检索结果展示来源时更清晰。

2.3 Stage ③ — Embedding Models（向量化模型）

将 chunk 文本转为高维向量。这是检索质量的核心决定因素，选型直接影响 recall 和 precision。

Class 名称功能说明你项目里对应核心参数 / 重要区别
OpenAIEmbeddings调用 OpenAI text-embedding-3 系列 API快速原型，数据不敏感model="text-embedding-3-large"；维度可裁剪；数据出境，注意合规
HuggingFaceEmbeddings本地 HuggingFace 模型（私有部署首选）Confluence 项目（内网）model_name="BAAI/bge-base-en-v1.5"；数据不外送；可 fine-tune
HuggingFaceBgeEmbeddingsBGE 系列专属封装，自动处理 query prefixBGE 模型优化封装自动在 query 前加 "Represent this sentence for searching..."；比直接用 HF 版精度高
OllamaEmbeddings本地 Ollama 模型，完全离线完全离线场景需要本地跑 Ollama 服务；model="nomic-embed-text"
CohereEmbeddingsCohere embed-v3，多语言强多语言场景input_type 区分 search_query vs search_document，混用会降精度
GoogleGenerativeAIEmbeddingsGoogle text-embedding-004GCP 生态task_type="retrieval_document" 或 "retrieval_query"；区分用途
FakeEmbeddings随机向量，仅用于测试单元测试 / CI不调用真实模型，加速测试；生产禁用
关键区别 — query 和 document 必须用不同 instruction：
BGE 系列：document 不加 prefix；query 加 "Represent this sentence for searching relevant passages:"
Cohere：用 input_type 参数区分。混用会让向量空间错位，检索精度显著下降。
Fine-tune embedding 模型：用 (用户问题, 正确template) 对做 contrastive learning，解决领域 semantic gap。

2.4 Stage ④ — Vector Stores（向量数据库）

存储 chunk 的向量和 metadata，支持相似度检索。生产选型要考虑：部署方式、metadata 过滤、实时更新、并发性能。

Class 名称功能说明你项目里对应核心参数 / 重要区别
FAISSFacebook 本地内存向量库原型开发，小规模save_local/load_local；不支持多进程、不支持实时单条更新；不适合生产
Chroma轻量嵌入式向量 DB本地部署，中小规模persist_directory 持久化；支持 metadata 过滤；单机部署简单
Qdrant高性能独立部署向量 DB（生产推荐）生产级私有部署REST + gRPC；支持 payload 过滤；支持实时 upsert；推荐替代 Pinecone
Pinecone云托管向量 DB，全托管无运维云端快速上线按量付费；数据在云端，金融场景注意合规风险
Weaviate多模态向量 DB图像+文本混合检索GraphQL 查询；BM25+向量 hybrid search 内置
PGVectorPostgreSQL pgvector 扩展已有 PG 的系统SQL + 向量一体；运维成本最低；connection_string 参数
ElasticsearchStoreES 向量 + 全文混合检索已有 ES 基础设施BM25 + 向量 hybrid 天然支持；企业内网常用
InMemoryVectorStore纯内存，进程内，无持久化单元测试仅测试用，进程结束数据消失
私有部署首选：Qdrant（独立进程，易横向扩展）或 PGVector（已有 PG 则零增量运维）。FAISS 不支持实时更新单条记录，Confluence 页面会被 Webhook 触发实时更新索引，FAISS 无法满足。

2.5 Stage ⑤ — Retrievers（检索器）

从 Vector Store 检索相关 chunk。这是 RAG 质量优化的核心战场，你的简历亮点都在这里。

Class 名称功能说明你项目里对应核心参数 / 重要区别
VectorStoreRetriever基础向量相似度检索默认检索search_type="similarity_score_threshold", score_threshold=0.7 过滤低质量结果
MultiQueryRetrieverLLM 改写多个 query 并行检索你 Confluence 项目有此解决 single-query 召回不全问题；自动 deduplicate；需要一个 LLM 做改写
ContextualCompressionRetriever检索后压缩/过滤低质量 chunk提升 context 质量配合 LLMChainFilter 或 EmbeddingsFilter 使用
EnsembleRetriever多路检索结果 RRF 算法融合Hybrid Search典型：BM25Retriever + VectorStoreRetriever；关键词+语义互补
BM25RetrieverBM25 关键词检索Hybrid Search 关键词路依赖 rank_bm25 库；与向量检索互补，专有名词召回强
ParentDocumentRetriever小 chunk 检索，返回 parent 大 chunk平衡精度与上下文完整小 chunk 精准定位，parent chunk 送给 LLM 保证上下文完整性
SelfQueryRetrieverLLM 从 query 提取 metadata 过滤条件结构化过滤+语义检索自动从"HR 部门的休假政策"提取 space_key='HR' 加入过滤
TimeWeightedVectorStoreRetriever时间加权，越新内容越靠前新鲜度要求高场景decay_rate 控制衰减速度；解决 Confluence 旧文档被高频引用的问题
你简历亮点对应关系：
"Multi-Query parallel retrieval" = MultiQueryRetriever
"relevance threshold filtering ensures answer quality" = score_threshold=0.7
"gracefully surfaces related page links when no match found" = threshold 未过时 fallback 返回 closest_pages 链接

2.6 Stage ⑥⑦ — LLM & Chat Models（生成模型）

LLM 是文本补全接口；ChatModel 是对话消息接口（推荐）。两者都是 Runnable，可以直接插入 LCEL 链。

Class 名称功能说明你项目里对应核心参数 / 重要区别
ChatGoogleGenerativeAIGemini 系列（你项目在用）EE Insights NLQ 项目model="gemini-2.0-pro"；支持 100 万 token 上下文；中国大陆访问受限
ChatOpenAIGPT-4o / GPT-4o-mini通用场景response_format={"type":"json_object"} 稳定 JSON 输出；mini 性价比高
ChatAnthropicClaude Sonnet / Opus复杂推理、代码生成max_tokens 必须设置；XML 格式输出更稳定
ChatOllama本地 Ollama 模型，完全离线内网离线场景model="qwen2.5:7b"；base_url 指向本地服务；数据不出内网
ChatDeepSeekDeepSeek V3 / R1 API性价比优先R1 有 reasoning_content 字段需特殊处理；API 价格远低于 GPT-4o
AzureChatOpenAIAzure OpenAI endpoint企业 Azure 合规部署azure_deployment + api_version 必须指定；数据留在 Azure 租户内

分离 prompt 模板逻辑与变量填充，支持复用和版本管理。

2.8 Output Parsers（输出解析器）

解析 LLM 输出为结构化数据。NLQ 参数提取的关键模块——LLM 输出 JSON，Parser 转为 Python 对象。

Class 名称功能说明你项目里对应核心参数 / 重要区别
StrOutputParser纯字符串直通（最简单）普通问答输出链式调用末尾常用；直接返回 LLM 文本
JsonOutputParser解析 JSON 格式输出NLQ 参数提取prompt 需明确要求输出 JSON；自动处理 markdown 代码块
PydanticOutputParser解析为 Pydantic 模型（类型安全）结构化参数提取get_format_instructions() 生成格式说明自动注入 prompt；类型验证+错误提示
CommaSeparatedListOutputParser解析逗号分隔列表Multi-query 改写结果简单场景无需 JSON；返回 Python list
XMLOutputParser解析 XML 格式输出Claude 模型（偏好 XML）Claude 系列用 XML 标签输出比 JSON 更稳定

2.9 Memory（对话记忆）

管理多轮对话历史，维持上下文连贯性。长对话需要在"完整性"和"token 消耗"之间权衡。

Class 名称功能说明你项目里对应核心参数 / 重要区别
ConversationBufferMemory存储全量对话历史短对话token 不设上限，长对话会超 context window
ConversationBufferWindowMemory只保留最近 K 轮控制 context 长度k=5 只保留最近 5 轮；简单但会丢失早期关键信息
ConversationSummaryBufferMemory近期全量 + 远期摘要（推荐生产用）长对话 RAG 场景max_token_limit 控制缓冲区大小；超限自动 LLM 摘要压缩
RedisChatMessageHistoryRedis 持久化对话历史多进程/节点共享session_id 隔离不同用户；ttl 控制过期时间
PostgresChatMessageHistoryPG 持久化对话历史已有 PG 基础设施connection_string 指定 DB；与 unmatched_queries 表同库管理

2.10 LCEL Chains & Callbacks（链式组装 + 回调追踪）

LCEL 用 | 运算符串联所有模块。Callbacks 拦截 pipeline 各阶段事件实现追踪。

Class 名称功能说明你项目里对应核心参数 / 重要区别
RunnableSequence ( | )LCEL 顺序链（现代写法，推荐）所有新项目prompt | llm | parser；类型安全；原生支持 async / stream / batch
RunnableParallel并行执行多个 RunnableMulti-query 并行检索dict 定义多分支，结果自动合并；{"context": retriever, "question": passthrough}
RunnablePassthrough透传输入不做修改保留原始 query在 RunnableParallel 中保留原始输入传给下游
RunnableLambda包装任意 Python 函数自定义处理步骤快速将任何函数插入 LCEL 链
AsyncCallbackHandler异步自定义回调基类自建 Splunk/DB trace重写 on_chain_end 异步写 trace，不阻塞主流程；你项目不用 Langfuse 的关键
StdOutCallbackHandler打印到标准输出本地调试开发阶段快速查看中间结果
RetrievalQA经典 RAG 链（旧 API）遗留代码已不推荐；用 LCEL 替代；面试如被问到说"知道但用 LCEL 替代了"
自建 trace 不用 Langfuse 的做法：继承 AsyncCallbackHandler，重写 on_chain_end，在里面异步写 PostgreSQL unmatched_queries 表或打结构化 JSON log 到 Splunk。no-match 判断在 on_retriever_end 回调里做——检查返回 chunks 数量和分数。
3. 完整 LCEL Pipeline 代码示例（Confluence Q&A 项目）
# ========== 离线阶段：批量索引（不在用户查询路径）========== from langchain_community.document_loaders import ConfluenceLoader from langchain_text_splitters import HTMLHeaderTextSplitter, RecursiveCharacterTextSplitter from langchain_community.embeddings import HuggingFaceEmbeddings from langchain_community.vectorstores import Qdrant loader = ConfluenceLoader(url=CONF_URL, username=USER, api_key=KEY, space_key="TEAM") splitter1 = HTMLHeaderTextSplitter(headers_to_split_on=[("h1","H1"),("h2","H2")]) splitter2 = RecursiveCharacterTextSplitter(chunk_size=512, chunk_overlap=50) embed = HuggingFaceEmbeddings(model_name="BAAI/bge-base-en-v1.5") store = Qdrant.from_documents(docs, embed, url=QDRANT_URL, collection_name="confluence") # ========== 在线阶段：用户查询（实时运行）========== from langchain.retrievers.multi_query import MultiQueryRetriever from langchain_core.prompts import ChatPromptTemplate from langchain_core.runnables import RunnableParallel, RunnablePassthrough from langchain_core.output_parsers import StrOutputParser from langchain_google_genai import ChatGoogleGenerativeAI # Stage ⑤：Multi-Query Retriever + 相似度阈值过滤 base_ret = store.as_retriever( search_type="similarity_score_threshold", search_kwargs={"score_threshold": 0.7, "k": 5} ) retriever = MultiQueryRetriever.from_llm( retriever=base_ret, llm=ChatGoogleGenerativeAI(model="gemini-2.0-flash") # 用 flash 改写 query，省成本 ) # Stage ⑥⑦：Augment + Generate（LCEL 链） prompt = ChatPromptTemplate.from_messages([ ("system", "你是 Confluence 知识库助手。仅基于以下内容回答，不要编造：\n\n{context}"), ("human", "{question}") ]) chain = ( RunnableParallel({"context": retriever, "question": RunnablePassthrough()}) | prompt | ChatGoogleGenerativeAI(model="gemini-2.0-pro", temperature=0) | StrOutputParser() ) # 调用 + 异步 trace 写入（不阻塞响应） result = await chain.ainvoke({"question": user_query}) asyncio.create_task(log_trace_async(user_query, result, ctx)) # ctx.has_answer=False 时 -> 写 PostgreSQL unmatched_queries # ctx.has_answer=True 时 -> 打结构化 JSON log 到 Splunk
4. 面试高频问题速查
如何提升检索召回率？
MultiQueryRetriever + EnsembleRetriever
多 query 改写并行检索解决单 query 召回不全；BM25 + 向量 RRF 融合弥补关键词匹配盲区（Hybrid Search）
如何过滤低质量检索结果？
score_threshold=0.7 + ContextualCompressionRetriever
相似度阈值过滤，低于阈值走 fallback 返回相关页面链接；Compression Retriever 进一步过滤 chunk 内无关内容
如何处理长文档保持上下文完整？
ParentDocumentRetriever
小 chunk（128 token）精准定位，检索命中后返回 parent 大 chunk（512 token）给 LLM，平衡检索精度和上下文完整性
如何实现多轮对话？
ConversationSummaryBufferMemory + MessagesPlaceholder
近期全量保留 + 远期 LLM 摘要压缩，MessagesPlaceholder 动态注入历史消息到 prompt
如何做结构化输出（NLQ 参数提取）？
PydanticOutputParser + JsonOutputParser
Pydantic 定义 schema，get_format_instructions() 自动生成格式说明注入 prompt；提供类型验证和自动错误提示
不用 Langfuse 怎么追踪 pipeline？
AsyncCallbackHandler（自定义）
继承基类重写 on_chain_end，异步写 trace 到数据库/Splunk，完全不阻塞主流程。no-match 写 DB，正常会话打 Splunk log
LCEL 和旧 Chain API 有什么区别？
RunnableSequence ( | ) vs RetrievalQA
LCEL 类型安全、原生 async/stream/batch 支持，更易调试；旧 Chain API（RetrievalQA、LLMChain）已不推荐，面试要知道"能识别但用 LCEL 替代"
Embedding 模型如何选型？
HuggingFaceEmbeddings vs OpenAIEmbeddings
私有部署/金融合规场景用 HF 本地（bge-base-en-v1.5），数据不外送；快速原型用 OpenAI；多语言用 bge-m3
如何解决 Confluence 索引实时性问题？
Webhook + 定时增量同步（双保险）
Webhook 事件驱动（延迟 <30s）为主；每小时定时增量同步（lastModified 过滤）为兜底，实现 at-least-once 语义
为什么选 fine-tune embedding 而非生成模型？
HuggingFaceEmbeddings + contrastive loss
用户问题和 SQL template 之间有 semantic gap，fine-tune bi-encoder 成本低（百条数据，单卡可跑）；生成模型 fine-tune 成本高但任务简单不值得
