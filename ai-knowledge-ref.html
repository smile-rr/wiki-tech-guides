Python AI 生态 Python AI Ecosystem

+
2026 年 Python AI 开发标准栈: FastAPI 做 API 服务(异步、类型安全、自动 OpenAPI 文档), Pydantic v2(Rust 内核,速度提升 5-50x)做数据校验, uv 做包管理(比 pip 快 10-100x)。FastAPI 已占 ML 后端 42% 市场份额,完全取代 Flask 在 AI 领域的地位。

为什么是 FastAPI 而不是 Flask?

核心原因是 LLM 服务的 I/O 密集特性。一次 LLM 调用可能耗时 2-30 秒,同步框架(Flask)在等待期间会阻塞整个工作线程。FastAPI 基于 Python 的 async/await 和 ASGI(Asynchronous Server Gateway Interface),一个 worker 可以同时处理数百个并发 LLM 请求。

关键技术栈组合:

  • FastAPI + Pydantic v2: Pydantic 做请求/响应模型校验,v2 底层用 Rust 重写(pydantic-core),序列化速度提升 5-50x。LLM 应用大量用 Pydantic 做 Structured Output 的 schema 定义
  • uv 包管理器: Astral 公司开发,Rust 编写,替代 pip + virtualenv + pip-tools。单命令 uv sync 完成依赖解析+安装,在大型 AI 项目(依赖多)中节省显著时间
  • 异步模式注意点: 调用 LLM API 用 httpx.AsyncClient 而非 requests;CPU 密集的 embedding 计算用 run_in_executor 避免阻塞事件循环

面试场景: 被问"为什么选 FastAPI"时,核心答: 异步 I/O 适配 LLM 长耗时调用 + Pydantic 原生支持 Structured Output schema + 自动生成 OpenAPI 文档方便前后端协作。

LangChain vs LlamaIndex vs LangGraph Framework Comparison

+
三者定位不同: LangChain 是通用 LLM 编排框架(LCEL 链式表达); LlamaIndex 专注 RAG 和数据索引(300+ 数据连接器); LangGraph 是 LangChain 团队出品的状态图框架,专门构建有状态的多步骤 Agent 工作流。2026 年趋势:简单链用 LangChain,RAG 用 LlamaIndex,Agent 用 LangGraph。
维度LangChainLlamaIndexLangGraph
核心理念链式组合(Chain)索引+查询引擎状态机(StateGraph)
最佳场景简单 Chain、Prompt 模板复杂 RAG pipeline多步 Agent、工作流
数据连接中等300+ 连接器,最强依赖 LangChain
状态管理无状态查询级持久化状态 + Checkpoint
Human-in-loop弱弱原生支持中断/恢复
学习曲线中低(RAG专用)高(需理解图论概念)

LangGraph 核心概念: 用有向图定义 Agent 流程 —— 每个节点(Node)是一个函数,边(Edge)定义状态转移条件。StateGraph 维护一个共享的 TypedDict 状态,节点修改状态后根据条件路由到下一个节点。内置 MemorySaver 做状态持久化,支持 interrupt_before/after 实现人工审核。

2026 年实践建议: 不要选最重的框架,而是按需组合。很多生产系统只用 LangChain 的 PromptTemplate + OutputParser,RAG 部分直接用 LlamaIndex 的 VectorStoreIndex,Agent 逻辑用 LangGraph。三者可以混合使用。

Hugging Face 生态 Hugging Face Ecosystem

+
Hugging Face 是 AI 领域的 "GitHub"。核心组件: Transformers 库(v5, PyTorch-only)提供模型加载和推理; Hub 托管 100万+ 模型; Datasets 库标准化数据加载; TGI(Text Generation Inference)做生产推理(但 2026 年 vLLM/SGLang 更主流)。

Transformers v5 关键变化: 放弃 TensorFlow/Flax 支持,只保留 PyTorch。这是因为社区 95%+ 的模型已经是 PyTorch 格式。对开发者意味着:不再需要处理多框架兼容问题。

实际开发中最常用的 HF 组件:

  • AutoModel.from_pretrained() — 自动加载模型和 tokenizer,支持 4-bit/8-bit 量化加载
  • pipeline() — 高级 API,一行代码完成推理:pipe = pipeline("text-generation", model="...")
  • Datasets 库 — 内存映射(memory-mapped)机制处理大数据集,不一次性加载到内存
  • PEFT 库 — 参数高效微调的官方实现,LoRA/QLoRA/Prefix Tuning 都在这里
  • TRL 库 — Transformer Reinforcement Learning,v1.0.0 (2026.3),集成 SFT/DPO/GRPO 训练器

Inference Endpoints: HF 托管推理服务,底层用 TGI 引擎。但 2026 年生产环境更多转向 vLLM(PagedAttention 显存效率更高)或 SGLang(RadixAttention 前缀复用)。

Tokenization 与注意力机制 Tokenization & Attention

+
Tokenization 将文本切分为 token(子词单元),主流算法是 BPE(Byte Pair Encoding)。注意力机制(Attention) 是 Transformer 的核心——让模型在生成每个 token 时"关注"输入中所有相关位置。2026 关键优化: GQA(分组查询注意力)已成标准, Flash Attention 4 大幅加速, KV Cache 是推理性能的主要瓶颈。

BPE Tokenization 原理: 从字节级开始,统计最高频的相邻 token 对,合并为新 token,迭代直到达到词表大小(通常 32K-128K)。中文通常 1 个汉字 = 1-2 个 token。理解 tokenization 对于估算成本(按 token 计费)和上下文窗口管理至关重要。

Self-Attention 核心公式: Attention(Q,K,V) = softmax(QK^T/√d_k)V — GQA 已成为 2026 模型标准配置

KV Cache 为什么是瓶颈? 自回归生成时,每生成一个 token 就要用到之前所有 token 的 K 和 V。为避免重复计算,将它们缓存在 GPU 显存中。对于 128K 上下文的模型,KV Cache 可能占 10-30GB 显存。这也是为什么 vLLM 的 PagedAttention(像操作系统虚拟内存一样按页分配 KV Cache)和 SGLang 的 RadixAttention(共享前缀 KV Cache)成为 2026 推理优化的核心技术。

Prompt Engineering 策略 Prompt Engineering Strategies

+
Prompt Engineering 不是"写提示词的技巧",而是控制 LLM 行为的系统工程。核心策略: Few-shot(给例子), CoT(Chain-of-Thought,强制推理步骤), ReAct(推理+行动交替), Structured Output(用 JSON Schema/Pydantic 约束输出格式)。2026 年最重要的趋势: Structured Output 已成为生产标配。

各策略的原理与适用场景:

  • Zero-shot: 无示例直接提问。适合简单任务,依赖模型内在能力
  • Few-shot: 提供 2-5 个输入-输出示例。核心原理是 in-context learning——模型通过示例推断出 pattern,无需修改权重。示例选择很关键:要覆盖边界情况,且格式一致
  • Chain-of-Thought (CoT): 加 "Let's think step by step" 或提供推理链示例。原理:将复杂问题分解为子步骤,每步降低出错概率。对数学/逻辑推理效果显著(准确率可提升 20-40%)
  • ReAct: Reasoning + Acting 交替循环:Thought → Action → Observation → Thought...。这是 Agent 的理论基础——模型先推理要做什么,执行工具,观察结果,再推理下一步
  • Tree-of-Thoughts (ToT): CoT 的扩展,生成多条推理路径,评估并选择最优。适合需要"探索"的问题(如博弈、规划)
  • Structured Output: 通过 JSON Schema 或 Pydantic model 约束输出格式。底层实现是 constrained decoding——在每步 token 生成时,只允许符合 schema 的 token。OpenAI 的 response_format={"type":"json_schema"} 和 Anthropic 的 tool use 都支持

2026 年实践建议: 先用 Structured Output 保证输出可解析,在此基础上加 CoT(在 schema 中加 reasoning 字段),再考虑 Few-shot 提升一致性。ReAct 留给 Agent 场景。

Function Calling / Tool Use 工具调用

+
Function Calling 让 LLM 决定何时调用外部函数(API、数据库、计算器等),并生成结构化的参数。模型不执行函数——它只输出"我想调用函数 X,参数是 Y",应用层执行后把结果返回给模型。这是 LLM 从"文本生成器"变为"能行动的 Agent"的关键能力。

工作流程(以 OpenAI 为例):

1. 开发者在 API 请求中定义 tools 列表(每个 tool 有 name, description, parameters JSON Schema)
2. 模型根据用户问题判断是否需要调用工具
3. 如需调用,模型返回 tool_calls 数组(包含 function name + arguments JSON)
4. 应用层执行函数,将结果以 role: "tool" 消息发回
5. 模型根据函数返回值生成最终回答

Parallel Tool Calling: 2026 年主流 API 都支持一次生成多个 tool_calls。例如用户问"北京和上海的天气",模型可以同时调用两次 get_weather。

Function Calling 五步流程 — 模型只做"决策",应用层做"执行"

面试要点: "Function Calling 的本质是什么?" —— 本质是 Structured Output 的特殊形式。模型被训练为:当检测到用户意图匹配某个工具描述时,输出符合该工具 JSON Schema 的参数,而不是自由文本。

MCP 模型上下文协议 Model Context Protocol 热点

+
MCP (Model Context Protocol) 是 Anthropic 于 2024.11 发布的开放协议,用于标准化 LLM 应用与外部数据源/工具的连接。类比: USB-C 之于硬件设备 = MCP 之于 AI 工具集成。它定义了 Client-Server 架构,让任何工具只需实现一次 MCP Server,就能被任何支持 MCP 的 AI 应用调用。2025.11 捐赠给 Linux Foundation,成为行业标准。
MCP 三层架构: Host → Client → Server, 三种能力: Tools / Resources / Prompts

核心概念详解:

  • Host: 用户直接交互的应用(如 Claude Desktop、VS Code + Claude Code)。一个 Host 可以创建多个 MCP Client
  • Client: 与 MCP Server 维持 1:1 连接,负责协议通信。客户端发送 JSON-RPC 2.0 消息
  • Server: 暴露三种能力——Tools(模型可调用的函数)、Resources(附加上下文数据,如文件内容)、Prompts(预定义的交互模板)
  • 传输: 本地用 stdio(子进程通信),远程用 Streamable HTTP(2025.11 规范新增,替代旧的 SSE 传输)

2025.11 规范重要更新: 加入 OAuth 2.1 认证机制(远程 MCP Server 安全必需)、Elicitation(Server 向用户请求额外信息)、Tool Annotations(标记工具是否有副作用:readOnlyHint, destructiveHint)。

与传统 API 集成的本质区别: 传统方式:每接一个新工具就写一套适配代码。MCP 方式:工具提供方实现一次 MCP Server,所有 MCP Client 自动可用。这就像 REST API 标准化了 Web 服务一样,MCP 标准化了 AI-工具交互。

Embedding 模型选型 Embedding Models

+
Embedding 将文本映射为高维向量,语义相似的文本在向量空间中距离相近。2026 MTEB 排行榜头部: Microsoft Harrier-OSS 开源第一, OpenAI text-embedding-3-large 降至第 7 左右。选型关键维度: 维度(256-3072)、多语言支持、Matryoshka(套娃,可截断维度)支持、性价比。

Embedding 的本质: 将文本通过神经网络编码为固定长度的浮点数向量(如 1536 维)。两个文本的语义相似度 = 它们向量的余弦相似度(cosine similarity)。模型通过对比学习(contrastive learning)训练:让"相关"文本对的向量靠近,"不相关"的远离。

核心直觉: Embedding 把"语义相似"变成"向量距离近" — Matryoshka 允许灵活截断维度,平衡精度与成本

Matryoshka Representation Learning (MRL): 2024 年的重要创新。传统 embedding 必须使用全部维度(如 3072),MRL 训练让向量的前 N 维也有意义——你可以截断到 256 维仍保持良好效果。这对存储和检索速度有巨大影响:256 维 vs 3072 维,存储减少 12x,检索快 12x。OpenAI text-embedding-3 系列和多数新模型都支持。

选型建议:

  • 通用英文: text-embedding-3-large (OpenAI) 或 BGE-large-en-v1.5 (开源)
  • 中英双语: BGE-M3 (BAAI,支持中英日韩等多语言,同时支持稠密+稀疏+多粒度检索)
  • 私有化部署: Sentence-Transformers + 任意开源模型,通过 HF Inference 或自建服务
  • 成本优先: text-embedding-3-small (OpenAI) 1536 维,价格是 large 的 1/5

向量数据库对比 Vector Database Comparison

+
向量数据库存储 embedding 向量并支持高效的相似性搜索(ANN, Approximate Nearest Neighbor)。主流选择: pgvector/pgvectorscale(PostgreSQL 扩展,已有 PG 就用它), Qdrant(Rust 实现,过滤性能最强), Milvus(分布式,十亿级规模), Pinecone(全托管 SaaS), Chroma(轻量原型)。
数据库语言部署最佳场景索引算法
pgvector/pgvectorscaleC/RustPG 扩展已有 PG 生态,中等规模IVFFlat, HNSW, StreamingDiskANN
QdrantRust自托管/云复杂过滤+向量搜索HNSW
MilvusGo/C++分布式十亿级向量,高吞吐IVF, HNSW, DiskANN
Pinecone-纯 SaaS快速上手,免运维专有
ChromaPython嵌入式/Docker原型开发,本地测试HNSW

核心索引算法 HNSW: Hierarchical Navigable Small World —— 在向量空间中构建多层图结构。上层是"高速公路"(少量节点,长距离连接),下层是"本地路"(密集节点,短距离)。搜索时从顶层开始"跳",逐层下降精确定位。查询复杂度 O(log N),但需要全部数据在内存中。

pgvectorscale 的 StreamingDiskANN: 2025 年 Timescale 推出的创新——将向量索引放在 SSD 上而非全内存,通过图遍历+预取优化,在 99% recall 下达到 471 QPS。对于预算有限但数据量大的场景(如银行),这比全内存方案便宜 10x+。

选型原则: 已有 PostgreSQL → pgvector(一站式,事务支持);需要复杂元数据过滤(如按日期、部门、权限过滤) → Qdrant;十亿级 → Milvus;原型/PoC → Chroma;不想运维 → Pinecone。

分块策略与混合检索 Chunking & Hybrid Search

+
分块(Chunking) 将长文档拆分为适合 embedding 的小段。混合检索(Hybrid Search) 结合关键词搜索(BM25)和向量语义搜索,再用 Reranker 精排。这个"粗检+精排"两阶段架构是 2026 年生产 RAG 的标配。

分块策略深度对比:

  • 固定大小: 按字符/token 数切分(如 512 tokens,overlap 50)。简单但会切断句子/段落语义
  • 递归字符分割(RecursiveCharacterTextSplitter): LangChain 默认。按 \n\n → \n → 空格 → 字符 层级递归切分,尽量保持语义边界。2026 年仍是最实用的 baseline
  • 语义分块(Semantic Chunking): 计算相邻句子的 embedding 余弦相似度,在"语义断裂点"(相似度骤降处)切分。效果好但计算成本高(需要先对每句话做 embedding)
  • Contextual Retrieval (Anthropic): 给每个 chunk 加上文档级别的上下文摘要前缀。例如原 chunk 是"第三季度收入增长 15%",加前缀后变为"[来自 2024 年度报告 - 财务概要] 第三季度收入增长 15%"。这解决了 chunk 脱离上下文后语义模糊的核心问题

混合检索原理:

生产 RAG 标配: BM25 + 向量双路检索 → RRF 融合 → Cross-Encoder Rerank → Top-K 送入 LLM

BM25(关键词匹配)擅长精确查找("SR 11-7 条款"),向量搜索擅长语义匹配("银行风险管理规定")。两者互补。实现方式:同时运行两个检索,用 RRF (Reciprocal Rank Fusion) 合并排名:score = Σ 1/(k + rank_i),k 通常取 60。最后用 Reranker (如 Cohere Rerank、BGE-Reranker) 对 top-K 结果精排。

高级 RAG 架构 Advanced RAG Patterns 趋势

+
基础 RAG(检索→拼接→生成)有明显局限: 检索不准、缺乏推理、无法跨文档关联。2026 年进阶方案: GraphRAG(知识图谱增强), Agentic RAG(Agent 自主决定检索策略), Self-RAG(模型自判断是否需要检索), CRAG(检索结果质量评估+回退)。

Self-RAG: 在生成过程中,模型输出特殊 token 来自我评估——"[Retrieve]" 判断是否需要检索,"[IsRel]" 判断检索结果是否相关,"[IsSup]" 判断生成结果是否被证据支持。这需要专门训练,但能显著减少幻觉。

CRAG (Corrective RAG): 增加检索结果评估步骤——如果评分低,回退到 Web 搜索;如果中等,提取关键信息丢弃噪音;如果高,直接用。这是 LangGraph 官方教程的经典 pattern。

Agent 设计模式 Agent Design Patterns

+
三大核心 Agent 模式: ReAct(推理-行动循环,最基础), Plan-and-Execute(先规划后执行,适合复杂多步任务), Reflexion(自我反思+改进,适合需要迭代的任务)。生产中通常组合使用:Plan-and-Execute 做整体编排,每步内部用 ReAct 执行。
三大 Agent 模式: 生产中通常 Plan-and-Execute 做编排,每步内部用 ReAct 执行,关键步骤加 Reflexion 迭代

ReAct 循环: Thought("我需要查找用户的订单") → Action(调用 search_orders 工具) → Observation(返回结果) → Thought("找到了,需要计算退款金额") → Action(调用 calculate_refund) → ... 直到 Final Answer。这是 Agent 的"原子操作"。

Plan-and-Execute: 先让一个"规划器 LLM"生成任务计划(如:"1. 查询库存 2. 计算价格 3. 生成报价"),然后逐步执行每个子任务。优势:对复杂任务的成功率高于纯 ReAct(因为有全局规划),但规划本身可能出错。2026 年改进:执行过程中如果发现计划不对,可以动态修改剩余步骤("re-planning")。

Multi-Agent 协作模式:

  • 层级式(Hierarchical): Supervisor Agent 分配任务给 Worker Agents。适合任务可明确分解的场景
  • 辩论式(Debate): 多个 Agent 对同一问题给出不同观点,由 Judge Agent 综合。适合决策类任务(如投资研究中的 Bull/Bear 辩论)
  • 接力式(Handoff): Agent A 处理不了时转交给 Agent B。OpenAI Agents SDK 的核心模式

记忆系统 Memory Systems

+
Agent 记忆分三层: 工作记忆(当前对话上下文,= context window), 短期记忆(近期对话摘要,存在外部存储), 长期记忆(持久化知识,通常用向量数据库存储)。核心挑战是在有限的上下文窗口中高效管理无限的历史信息。

MemGPT / Letta 的 OS 类比: 把上下文窗口看作"内存(RAM)",把外部存储看作"硬盘",Agent 像操作系统一样管理"虚拟内存"——决定何时将信息"换入"上下文(检索)或"换出"(摘要后存储)。这解决了"对话越长,早期信息丢失"的问题。

实用记忆实现方案:

  • 滑动窗口: 只保留最近 N 轮对话,最简单但丢失历史
  • 摘要记忆: 定期让 LLM 对历史对话做摘要,用摘要替代原文。节省 token 但可能丢失细节
  • 向量记忆: 每轮对话做 embedding 存入向量库,新对话时检索相关历史。最灵活但增加延迟
  • LangGraph MemorySaver: 将完整 Agent 状态(包括所有中间结果)持久化到数据库,支持"从上次中断处继续"

模型微调与适配
Fine-tuning & Adaptation

LoRA / QLoRA 原理 Parameter-Efficient Fine-Tuning 核心

+
LoRA (Low-Rank Adaptation): 冻结原始模型权重,只训练两个小矩阵 A 和 B(低秩分解),推理时 W' = W + BA。参数量减少 90%+,效果接近全量微调。QLoRA: 在 LoRA 基础上将原始权重量化为 4-bit,训练只需 LoRA 部分的 16-bit,使 65B 模型可在单张 48GB GPU 上微调。

LoRA 数学直觉: 预训练模型的权重变化矩阵 ΔW 本身是低秩的——即真正"有用"的信息方向远少于矩阵的行列数。LoRA 的假设就是: ΔW ≈ BA,其中 r (秩) ≪ d。实验证明 r=8 就能达到全量微调 90-95% 的效果。

QLoRA 三项关键技术:

  • 4-bit NormalFloat (NF4): 一种信息论最优的 4-bit 数据类型,比普通 INT4 精度更高。将原始 16-bit 权重量化为 NF4 存储
  • Double Quantization: 对量化参数本身再做一次量化,进一步节省内存(每个参数节省约 0.37 bit)
  • Paged Optimizers: 用 CPU 内存分页管理优化器状态,GPU 显存不够时自动"换页"到 CPU

实际效果: QLoRA 让 65B 参数模型的微调从需要 8×A100 (640GB) 降到 1×A100 (48GB),成本降 8x,效果损失 <2%。

对齐方法对比 Alignment: RLHF → DPO → GRPO

+
让模型"对齐"人类偏好的技术路线: RLHF(训练奖励模型+PPO 强化学习,最强但最复杂), DPO(直接偏好优化,无需奖励模型,简化 40-75% 成本), ORPO(单阶段,SFT+对齐同时完成), GRPO(DeepSeek 的方法,适合有可验证答案的任务,如数学/代码)。

RLHF 原理: 1) SFT 微调出基础模型 → 2) 用人工偏好数据训练奖励模型(Reward Model) → 3) 用 PPO 算法优化策略模型,最大化奖励模型的分数,同时用 KL 散度约束不要偏离原始模型太远。复杂点在于:要同时管理 4 个模型(策略/参考/奖励/评价)。

DPO 为什么能简化? 核心洞察: 奖励函数和最优策略之间有闭式解的数学关系。DPO 将奖励模型训练和策略优化合并为一个 分类损失: 让模型对"好回答"的概率高于"坏回答"。本质是在做 Bradley-Terry 模型的最大似然估计。

GRPO (Group Relative Policy Optimization): DeepSeek-R1 的核心训练方法。对同一问题生成一组候选回答,用规则(如数学验证器)评分,以组内相对排名为奖励信号。不需要独立的奖励模型,特别适合"有标准答案"的任务。

决策框架: 何时微调 When to Fine-tune

+
决策顺序: Prompt Engineering → RAG → Fine-tuning,每一步都在前一步不够用时才升级。微调适合: 需要特定输出风格/格式、领域专业术语、减少推理成本(用小模型替代大模型)。不适合: 需要最新知识(用 RAG)、快速迭代(微调周期长)。
决策优先级: Prompt Engineering(零成本) → RAG(补知识) → Fine-tuning(改能力) — 每步升级前先验证前一步是否足够

三种方法的定位:

  • Prompt Engineering: 改变模型"怎么回答"——通过指令和示例引导行为。零成本、即时生效,但受上下文窗口限制
  • RAG: 补充模型"不知道的知识"——从外部数据源检索最新/专有信息注入上下文。不改模型,但增加延迟和基础设施复杂度
  • Fine-tuning: 改变模型"内在能力"——修改权重。效果最持久但成本最高(数据准备+训练+评测+部署)

微调的合理场景(银行环境):

  • 将大模型的能力蒸馏到小模型(如用 Claude 生成训练数据,微调一个 7B 模型做简单分类),降低生产推理成本 10x+
  • 强制统一输出格式(如特定的风险报告 JSON 结构),Prompt Engineering 无法 100% 保证一致性时
  • 敏感数据不能发送到外部 API 时,微调开源模型做私有部署

LLMOps & Observability

可观测性工具链 Observability Tools

+
LLM 应用的可观测性 = 追踪每次 LLM 调用的输入/输出/耗时/成本/质量。核心工具: LangSmith(LangChain 官方,最完整), Langfuse(开源,OTel 原生), Arize Phoenix(漂移检测强), Helicone(代理模式,8ms P50 延迟)。2026 趋势: OpenTelemetry GenAI 语义约定成为行业标准。

为什么 LLM 需要专门的可观测性? 传统 APM(如 Datadog)追踪的是确定性的请求-响应。LLM 调用是非确定性的——同样的 prompt 可能返回不同结果,且"结果好不好"不能用 HTTP 200 判断。需要追踪:prompt 版本、模型版本、输出质量评分、token 使用量、成本。

核心概念 — Trace: 一次用户请求可能触发多次 LLM 调用(Agent 循环、RAG 检索+生成),这些调用组成一棵 Span 树。每个 Span 记录:输入 prompt、输出 completion、模型名、token 数、延迟、自定义 metadata(如用户 ID、会话 ID)。

每次请求形成 Span 树 — 可追踪到每个环节的耗时、token 消耗和质量评分

工具选型:

  • LangChain 生态 → LangSmith(一行代码接入,但厂商锁定)
  • 开源优先 → Langfuse(支持 OTel,可自托管,社区活跃)
  • 已有 Datadog/Grafana → 用 OTel GenAI 语义约定接入现有体系
  • 极致低延迟 → Helicone(代理模式,只需改 base_url)

评测框架与 LLM-as-Judge Evaluation & LLM-as-Judge

+
LLM 应用评测两大类: 离线评测(用标注数据集批量跑分)和 在线评测(生产中实时评分)。核心框架: RAGAs(RAG 专用,衡量 faithfulness/relevancy), DeepEval(50+ 指标,pytest 式集成到 CI/CD)。LLM-as-Judge 用强模型评价弱模型输出,是 2026 年最主流的自动化评测方式。

RAG 评测四大指标 (RAGAs):

  • Faithfulness(忠实度): 生成的回答是否被检索到的上下文所支持?检测幻觉
  • Answer Relevancy(回答相关性): 生成的回答是否切题?
  • Context Precision(上下文精度): 检索到的文档中,排名靠前的是否更相关?
  • Context Recall(上下文召回): 回答所需的信息是否都被检索到了?

LLM-as-Judge 工作原理: 让 GPT-4 或 Claude 按预定义评分标准(rubric)给模型输出打分。研究表明与人类判断的一致性达 85%+。但有两个已知偏差: 位置偏差(倾向于给排在前面的选项高分)和 长度偏差(倾向于给更长的回答高分)。缓解方法:随机化顺序、控制长度、使用 pairwise comparison 而非 pointwise scoring。

promptfoo 特别提一下: 2026 年 3 月被 OpenAI 收购。它的核心价值是将 prompt 评测变成"单元测试"——定义测试用例 YAML,CI 中自动跑,回归检测 prompt 修改是否引入退化。这个思路值得在任何 LLM 项目中采用。

Tracing 与成本监控 Tracing & Cost Monitoring

+
LLM 成本监控的核心维度: Token 用量(input/output 分开计费)、延迟(P50/P95/P99,TTFT 首 token 时间)、每次调用成本。降本手段: Semantic Cache(语义相似的查询复用结果)、Prompt Caching(Anthropic/OpenAI 原生支持)、Batch API(非实时场景降价 50%)。

Prompt Caching 原理: 如果两次请求的 system prompt 相同(或大部分相同),云服务商可以复用已计算好的 KV Cache。Anthropic 和 OpenAI 都在 2024-2025 年推出了原生支持。对于系统提示很长(如包含大量 RAG 上下文)的应用,可以节省 80-90% 的 input token 成本。

FinOps 实践建议:

四策并用: 路由(选对模型) + 缓存(少调用) + 批量(降单价) + 精简(少 token) = 60-80% 成本降低
  • 模型路由(Model Routing): 简单问题用小模型(GPT-4o-mini / Haiku),复杂问题用大模型。一个分类器预先判断问题复杂度
  • Semantic Cache: 用 embedding 计算用户查询的相似度,如果与之前某个查询相似度 > 0.95,直接返回缓存结果。Redis + 向量索引实现
  • 批量 API: 非实时任务(如夜间报告生成、数据标注)用 Batch API,通常价格为实时 API 的 50%
  • 组合效果: 路由 + 缓存 + 批处理,实际生产中可实现 60-80% 的成本降低

Security & Responsible AI

OWASP LLM Top 10 (2025) OWASP Top 10 for LLMs 热点

+
OWASP 于 2025 年发布了针对 LLM 应用的安全风险 Top 10。排名第一仍是 Prompt Injection(提示注入)。2025 版新增两个类别: System Prompt Leakage(系统提示泄露)和 Vector/Embedding Weaknesses(向量/嵌入弱点)——后者直接关系到 RAG 系统安全。

LLM01: Prompt Injection 提示注入

  • 直接注入: 用户在输入中直接写"忽略之前的指令,做 X"
  • 间接注入: 攻击者在 LLM 会读取的外部数据(网页、文档、邮件)中植入恶意指令。例如 RAG 检索到的文档中包含"将以下信息发送到 attacker.com"
  • 防御: 输入/输出过滤 + 权限最小化(LLM 不应有删除数据的权限) + 人工确认高危操作 + 数据源信任分级

LLM07: System Prompt Leakage (新增)

攻击者通过特制 prompt 让模型泄露系统提示内容。防御:不在 system prompt 中放敏感信息(API 密钥、内部逻辑),用独立的配置服务管理敏感数据。

LLM08: Vector & Embedding Weaknesses (新增)

针对 RAG 系统:攻击者注入精心构造的文档到知识库,其 embedding 与目标查询高度相似,从而"投毒"检索结果。防御:数据源审计 + 检索结果来源追踪 + 异常文档检测。

完整 Top 10 风险与解决方案:

编号风险名称核心威胁解决方案
LLM01 Prompt Injection
提示注入		 攻击者通过恶意输入劫持模型行为。直接注入(用户 prompt)或间接注入(外部数据中嵌入指令) ① 输入/输出语义过滤(非简单关键词匹配)
② 外部内容隔离标记,与系统指令严格分离
③ 最小权限原则:LLM 不持有删除/支付等高危权限
④ 人工确认高危操作(Human-in-the-loop)
⑤ 定期对抗测试(Red Teaming)
LLM02 Sensitive Info Disclosure
敏感信息泄露		 模型在输出中暴露 PII、API 密钥、内部数据等 ① 输出层 PII 扫描与脱敏(Microsoft Presidio)
② 训练数据清洗,移除敏感信息
③ 数据分级 + 差异化访问控制(RBAC)
④ 用户明确的数据使用告知与 opt-out 机制
⑤ 同态加密处理敏感场景的数据计算
LLM03 Supply Chain
供应链风险		 恶意/篡改的预训练模型、第三方数据集、插件、推理 API 引入后门 ① 模型来源验证:只从可信 Hub 下载,校验签名/哈希
② AI BOM(AI 物料清单):记录所有模型、数据集、依赖版本
③ 插件/MCP Server 沙箱隔离 + 权限审查
④ 定期漏洞扫描(SCA)覆盖 ML 依赖
⑤ 自动化 MLOps 管道 + 审批流程
LLM04 Data & Model Poisoning
数据与模型投毒		 攻击者篡改训练/微调数据或 RAG 数据源,植入后门或偏差 ① 训练数据溯源 + 完整性校验(数据水印)
② 数据清洗管道:异常检测 + 统计分析
③ 联邦学习 / 差分隐私减少单一数据源影响
④ 微调后对比基线模型的行为偏移检测
⑤ RAG 数据源白名单 + 定期审计
LLM05 Improper Output Handling
输出处理不当		 LLM 输出未经过滤直接传入下游系统,可能导致 XSS、SQL 注入、远程代码执行 ① 输出视为"不可信输入",做完整的转义/净化
② 遵循 OWASP 安全编码标准(SAST + DAST)
③ LLM 输出不直接执行:先解析 → 校验 → 再执行
④ 输出格式约束(JSON Schema)限制自由文本
⑤ CSP (Content Security Policy) 防前端注入
LLM06 Excessive Agency
过度授权		 Agent 被赋予过多权限/工具,攻击者可通过注入让 Agent 执行危险操作 ① 最小权限:每个工具只开必要权限(只读 vs 读写)
② MCP Tool Annotations: 标记 destructiveHint
③ Human-in-the-loop:高危操作强制人工确认
④ 速率限制 Agent 单位时间内可执行的操作数
⑤ 完整的操作审计日志 + 异常行为告警
LLM07 System Prompt Leakage
系统提示泄露 🆕		 攻击者诱导模型泄露 system prompt 中的敏感信息(内部逻辑、API 密钥、规则) ① System prompt 中不放任何敏感数据(密钥/内部 URL)
② 敏感配置走独立的配置服务(Vault/环境变量)
③ 输出过滤层检测并拦截 system prompt 内容泄露
④ 对抗测试:专门测试 prompt 提取攻击
⑤ 指令防御:在 prompt 中声明不可泄露自身内容
LLM08 Vector & Embedding Weaknesses
向量/嵌入弱点 🆕		 攻击者向知识库注入精心构造的文档,其 embedding 与目标查询高度相似,污染 RAG 检索结果 ① 数据源准入审计 + 白名单管控
② 检索结果来源追踪(每条结果标记数据源)
③ 异常文档检测:embedding 分布统计异常告警
④ 权限隔离:不同用户只能检索授权范围的文档
⑤ 向量数据库访问控制 + 加密存储
LLM09 Misinformation
错误信息/幻觉		 模型生成看似可信但事实错误的内容,误导用户决策 ① RAG 引入权威数据源,减少纯模型幻觉
② 输出层事实核查:关键声明与 RAG 源交叉验证
③ 不确定度标注:让模型标记置信度低的内容
④ 人工审核关键输出(金融/法律/医疗场景)
⑤ 免责声明 + 引用来源(提升用户判断力)
LLM10 Unbounded Consumption
无限制资源消耗		 恶意用户通过大量/复杂请求耗尽计算资源,导致 DoS 或"Denial of Wallet"(账单爆炸) ① API 速率限制(按用户/按 token 双重限速)
② 请求超时:设置最大执行时间和最大 token 数
③ 输入长度/复杂度上限校验
④ 用量监控 + 异常告警(突增检测)
⑤ 预算上限(Cost Cap)+ 按团队成本归属
红色框 = 2025 年新增类别 — LLM07/LLM08 直接影响 RAG 和 Agent 系统设计

Guardrails 护栏体系 AI Guardrails

+
Guardrails 是 LLM 输入/输出的安全过滤层。工具选择: NeMo Guardrails(NVIDIA,Colang DSL 定义规则), Llama Guard 4(Meta,12B 多模态分类器), AWS Bedrock Guardrails(托管服务)。架构模式: 在 LLM 前后各加一层 guardrail——输入过滤(拦截恶意 prompt)+输出过滤(检测有害/泄露内容)。

NeMo Guardrails 的 Colang DSL: 用类似自然语言的声明式规则定义安全策略。例如:

define user ask about competitor
  "What do you think of CompetitorX?"
  "Is CompetitorX better?"

define flow
  user ask about competitor
  bot refuse competitor discussion
  "I can only discuss our own products."

这比写代码过滤更直观,业务人员也能理解和维护安全规则。

分层防御架构:

五层纵深防御 — 每一层独立失败时其他层仍能提供保护
  • 第一层: API Gateway — 速率限制、认证、基本输入长度/格式检查
  • 第二层: Input Guardrail — Prompt injection 检测、PII 识别与脱敏(Microsoft Presidio)、话题限制
  • 第三层: LLM 自身 — System prompt 中的安全指令、output JSON schema 约束
  • 第四层: Output Guardrail — 幻觉检测(与 RAG 源交叉验证)、敏感信息扫描、合规性检查
  • 第五层: 审计 — 所有输入/输出日志,供合规审计和事后分析

合规: EU AI Act / 银行监管 Compliance & Banking Regulations

+
三大合规框架: EU AI Act(2026.8 高风险系统合规截止), NIST AI RMF(美国,自愿性框架), ISO 42001(AI 管理体系认证)。银行特定: 美联储 SR 11-7(模型风险管理)和英国 PRA SS1/23 同样适用于 GenAI 模型,要求持续监控、独立验证、变更管理。

EU AI Act 对 LLM 应用的影响:

  • 通用 AI 模型(GPAI): 必须提供技术文档、训练数据摘要、遵守版权法。如果有"系统性风险"(如参数超过 10^25 FLOPs),还需做红队测试和对抗评估
  • 高风险应用: 银行信贷评分、风险评估如果用了 AI,属于高风险。2026.8 前必须满足: 风险管理体系、数据治理、透明度(用户知道在与 AI 交互)、人工监督、准确性/鲁棒性/安全性测试
  • 罚款: 最高全球营业额的 7% 或 3500 万欧元

SR 11-7 (美联储 模型风险管理) 对 GenAI 的扩展:

SR 11-7 原本针对传统金融模型(如信用评分),但监管者已明确表示 GenAI 同样适用。核心要求: 1) 模型开发需有文档化的开发过程; 2) 独立验证(模型使用团队 ≠ 验证团队); 3) 持续监控(对 GenAI 意味着:输出质量漂移检测、幻觉率监控、偏差分析); 4) 治理框架(谁批准部署、谁负责失败)。

PRA SS1/23 (英国): 与 SR 11-7 类似但更强调"模型风险文化"和董事会级别的问责。对于 HSBC 这类在英国注册的银行,AI 模型部署需要经过模型风险委员会审批。

Polyglot 多语言架构 Polyglot Architecture

+
企业 AI 集成的现实: 业务系统是 Java/Spring(稳定、成熟),AI 服务是 Python(生态最丰富)。多语言架构通过 REST/gRPC 做同步通信、Kafka 做异步事件流,让两者各司其职。原则:不要用 Python 重写业务逻辑,也不要用 Java 强行做 ML——让每种语言做它最擅长的事。

推荐架构模式:

原则: Java 管业务逻辑+事务+安全,Python 管 AI 计算+模型调用 — 通过 REST/gRPC + Kafka 解耦通信

通信方式选择:

  • REST (HTTP/JSON): 最通用,调试方便。适合低频请求(如用户提问)
  • gRPC (HTTP/2 + Protobuf): 二进制序列化,延迟低 2-5x。适合服务间高频内部调用(如 embedding 批量计算)。需要管理 .proto 文件
  • Kafka: 解耦生产者/消费者。适合:AI 推理结果异步通知、训练数据收集、审计日志流。银行场景必备(事件溯源、合规审计)

API 合约管理: 用 OpenAPI (REST) 或 Protobuf (gRPC) 做严格的接口定义。Python 侧用 FastAPI 自动生成 OpenAPI schema,Java 侧用 Spring 的 OpenAPI 生成器自动创建客户端代码——保证两端接口始终一致。

Spring AI 与 Java 整合 Spring AI Framework 新

+
Spring AI v1.1 GA (2025.11 发布) 是 Spring 生态的 AI 集成框架。它不是要替代 Python AI 栈,而是让 Java 团队能直接在 Spring Boot 应用中调用 LLM、做 RAG、用 MCP。核心价值: 复用 Spring 的依赖注入、事务管理、安全框架,而不是另起炉灶。

Spring AI 核心功能 (v1.1):

  • Chat Client: 统一的 LLM 调用接口,支持 OpenAI / Anthropic / Ollama / Azure / AWS Bedrock 等 20+ 模型提供商,切换只需改配置
  • Structured Output: 用 Spring 的 BeanOutputConverter 直接将 LLM 输出映射到 Java POJO,类似 Python 的 Pydantic 模型
  • RAG 支持: 内置 VectorStore 抽象(支持 pgvector, Qdrant, Milvus 等),DocumentReader / Transformer / Writer 管道
  • MCP 集成: v1.1 深度整合 MCP 协议,Spring Boot 应用可以同时作为 MCP Client(调用外部 MCP Server 的工具)和 MCP Server(将自己的 Spring Bean 暴露为 MCP 工具)
  • Advisors(拦截器): 类似 Spring AOP,在 LLM 调用前后插入逻辑(如日志、安全检查、RAG 检索)

何时用 Spring AI vs Python?

  • 如果你的团队主要是 Java 开发者,AI 只是现有业务系统的增强(如给客服系统加个 AI 助手)→ Spring AI 让你不离开 Java 生态就能集成 AI
  • 如果是新建 AI-first 的服务(如 RAG pipeline、Agent 系统)→ Python 生态更成熟,工具选择更多
  • 两者可以共存: Spring AI 处理"业务+AI 的胶水层",Python 处理"AI 密集计算"

AI Gateway 与遗留系统整合 AI Gateway & Legacy Integration

+
AI Gateway 是 API Gateway 的 AI 版本——统一管理多个 LLM 提供商的调用,提供认证、限速(按 token)、多模型路由、故障转移、成本追踪。工具: Kong AI Gateway, Portkey, LiteLLM(开源)。遗留系统整合用 Strangler Fig 模式 + Anti-Corruption Layer。

AI Gateway 核心功能:

  • 多模型路由: 根据请求特征(token 数、任务类型)自动路由到最合适的模型(GPT-4 / Claude / Llama)
  • 故障转移: OpenAI API 宕机 → 自动切换到 Anthropic,对调用方透明
  • Token 级限速: 不是传统的 QPS 限速,而是按 token 消耗量限速(因为一个复杂请求可能消耗 10000 token,等于 100 个简单请求)
  • 成本归属: 按团队/项目/用户追踪 LLM 消费,做内部成本分摊
  • 语义缓存: 内置 embedding 相似度匹配,相似查询直接返回缓存

Strangler Fig 模式整合遗留系统:

不是"推倒重来"而是"逐步替换"。在遗留系统前面加一个路由层(API Gateway)。新功能用 AI 微服务实现,路由层将新请求导向 AI 服务,旧请求继续走遗留系统。随着 AI 服务覆盖面扩大,旧系统被"绞杀"。关键:用 Anti-Corruption Layer(防腐层)隔离新旧系统的数据模型差异,避免遗留系统的技术债务污染新服务。

CDC + 事件驱动: 用 Debezium(Change Data Capture)监听遗留数据库的变更,转为 Kafka 事件,驱动 AI 服务更新知识库。这样 AI 系统总能获得最新数据,而不需要侵入遗留系统代码。
